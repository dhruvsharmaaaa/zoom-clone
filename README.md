# Zoom Clone With End-to-End Features

A functional video-conferencing web app that replicates Zoom's core dashboard
and meeting-room workflows: instant meetings, scheduled meetings, join by ID
or link, real peer-to-peer video/audio via WebRTC, and host controls.

## Tech Stack

| Layer      | Technology                                             |
|------------|---------------------------------------------------------|
| Frontend   | Next.js 14 (App Router), React 18, Tailwind CSS         |
| Backend    | Python, FastAPI, WebSockets (signaling)                 |
| Database   | SQLite via SQLAlchemy ORM                                |
| Video/Audio| WebRTC (peer-to-peer mesh), STUN via Google's public server |

## Database Schema

```
Meeting                          Participant
--------                         -----------
id (PK)                          id (PK)
meeting_code (unique)            meeting_id (FK -> Meeting.id)
title                            display_name
description                      is_host
host_name                        is_muted
meeting_type (instant/scheduled) is_video_on
status (scheduled/active/ended)  is_active
scheduled_time                   joined_at
duration_minutes                 left_at
created_at / started_at / ended_at
```

One `Meeting` has many `Participant` rows (one-to-many). This keeps the
schema normalized: host actions like "mute all" or "remove participant" are
simple `UPDATE`/`SELECT` queries against the `participants` table instead of
mutating a nested JSON blob.

## Setup Instructions

### Prerequisites
- Python 3.10+
- Node.js 18+
- npm

### 1. Backend

```bash
cd backend
python -m venv venv
source venv/bin/activate        # Windows: venv\Scripts\activate
pip install -r requirements.txt
python -m app.seed               # creates zoom_clone.db and inserts sample data
uvicorn app.main:app --reload --port 8000
```

Backend now runs at `http://localhost:8000`. Interactive API docs (auto
generated by FastAPI) are at `http://localhost:8000/docs`.

### 2. Frontend

Open a **second terminal** (keep the backend running in the first):

```bash
cd frontend
npm install
cp .env.example .env.local       # already points at localhost:8000 by default
npm run dev
```

Frontend now runs at `http://localhost:3000`.

### 3. Try it

- Open two browser windows/tabs at `http://localhost:3000`.
- In one, click **New Meeting**, enter a name, start it.
- Copy the meeting link/ID and open it in the second window — enter a
  different name and join.
- You should see live camera video between both windows (grant camera/mic
  permission when prompted). Try mute, camera off, and (as host) Mute All /
  remove participant from the Participants panel.

## Assumptions

- **No login/auth**: a default user ("Dhruv Sharma") is assumed logged in
  everywhere, per the assignment's instructions. There is no password or
  session management.
- **Host = first joiner**: whoever is the first active participant in a
  meeting is treated as the host for that session (mirrors how a Zoom
  meeting "starts" when the host joins).
- **Mesh WebRTC**: video connects participants directly to each other
  (peer-to-peer), which is correct and simple for small meetings. Large
  meetings (10+ people) would need a media server (SFU) in a production
  system — out of scope here.
- **Screen share / chat / recording** are shown in the UI as visual
  placeholders (disabled buttons) to match Zoom's layout, but are not
  functionally implemented — they weren't in the "Must Have" list.
- **SQLite** is used as required; the same SQLAlchemy models would work
  against Postgres/MySQL with only the connection string changed.

## Deployment

- **Frontend**: deploy to Vercel (`vercel.com`) — connect the GitHub repo,
  set root directory to `frontend`, add env var `NEXT_PUBLIC_API_URL`
  pointing at your deployed backend URL.
- **Backend**: deploy to Render (`render.com`) as a Web Service — root
  directory `backend`, build command `pip install -r requirements.txt`,
  start command `uvicorn app.main:app --host 0.0.0.0 --port $PORT`.
  Note: Render's free tier disk is ephemeral, so the SQLite file resets on
  redeploy — fine for a demo/assignment.

See `DEPLOYMENT.md` for the full step-by-step guide.
